# 客户端对接说明书

本文档面向 `cline` 或其他外部客户端接入方，只说明接入时必须遵守的规则，不解释服务端内部实现。

## 1. 连接规则

- 通信方式：TCP。
- 客户端连接到服务端提供的 IP 和 Port。
- 同一局对战需要两个客户端连接到同一个服务端。
- 先连接成功的一方为红方。
- 后连接成功的一方为黑方。

## 2. 棋盘坐标规则

棋盘使用网格坐标：

| 字段 | 含义 | 范围 |
| --- | --- | --- |
| `row` | 行 | `0..9` |
| `col` | 列 | `0..8` |

坐标规则：

- 左上角是 `(0,0)`。
- `row` 从上到下递增。
- `col` 从左到右递增。
- 右下角是 `(9,8)`。

## 4. 报文格式

### 4.1 客户端发给服务端

长度固定为 3 字节：

| 字节 | 含义 |
| --- | --- |
| 第 1 字节 | `id` |
| 第 2 字节 | `row` |
| 第 3 字节 | `col` |

格式：

```text
[id][row][col]
```

含义：请求将 `id` 对应的棋子移动到 `(row,col)`。

### 4.2 服务端回给发送方的结果

长度固定为 1 字节：

| 值 | 含义 |
| --- | --- |
| `1` | 本次请求成功 |
| `-1` | 本次请求失败 |

补充规则：

- 客户端每收到一次 `-1`，记为 1 次违规。
- 同一方累计 3 次违规，立即判负。

### 4.3 服务端同步给对手方

长度固定为 3 字节：

```text
[id][row][col]
```

含义：通知对手方，这一步已经被服务端接受并生效。

### 4.4 服务端终局通知

当对局结束时，服务端会向双方各发送 1 个终局通知包。

长度固定为 2 字节：

| 字节 | 含义 |
| --- | --- |
| 第 1 字节 | 固定为 `0xFD` |
| 第 2 字节 | 终局结果码 |

结果码定义：

| 值 | 含义 |
| --- | --- |
| `1` | 红方胜 |
| `2` | 黑方胜 |
| `3` | 和棋 |

## 5. 收发流程

客户端按下面流程接入即可：

1. 连接服务端的 IP 和 Port。
2. 根据连接顺序确认自己是红方还是黑方。
3. 在轮到自己时，发送 3 字节请求 `[id][row][col]`。
4. 等待服务端返回 1 字节结果。
5. 如果结果为 `1`，说明本次走子成功。
6. 如果结果为 `-1`，说明本次走子失败，本地不要生效，同时该方违规次数加 1。
7. 同一方累计 3 次违规后，服务端判该方负。
8. 当自己是被动接收方时，等待服务端同步 3 字节 `[id][row][col]`，并按该结果更新本地棋盘。
9. 如果服务端随后发来终局通知 `[0xFD][result]`，说明本局已经结束，客户端应停止继续发送普通走子请求。

收包时序说明：

- 发送方合法走子：先收 `1`，若本步导致终局，再继续收 `[0xFD][result]`。
- 对手方合法走子：先收 `[id][row][col]`，若本步导致终局，再继续收 `[0xFD][result]`。
- 发送方非法走子：先收 `-1`，若该失败触发终局，再继续收 `[0xFD][result]`。

接入建议：

- TCP 是字节流，客户端应按固定长度依次读取 `ACK / 同步 / 终局通知`，不要假设一次 `recv()` 只返回一个完整业务消息。

## 6. 客户端必须遵守的规则

客户端发起请求时，必须遵守以下规则：

- `id` 必须来自本文的固定对照表。
- `row` 必须在 `0..9`。
- `col` 必须在 `0..8`。
- 只能在轮到自己时发送请求。
- 只能操作己方棋子。
- 不能移动已经被吃掉的棋子。
- 不能吃己方棋子。
- 棋子的走法必须符合中国象棋规则。
- 不能通过走子让己方将/帅处于被攻击状态。
- 同一方累计 3 次违规会直接判负。

服务端还会按以下终局规则裁定结果：

- 一方将/帅被吃，另一方胜。
- 一方累计 3 次违规，另一方胜。
- 一方无合法着法，另一方胜。
- 长将判负。
- 长捉判负。
- 普通重复局面判和。

如果不满足上述规则，服务端会返回失败结果。

## 7. 长将 / 长捉 / 重复局面判定口径（当前服务端实现）

当前服务端按以下口径处理重复局面：

- 只有当“同一局面且轮到同一方走”重复出现至少 3 次时，才进入重复局面判定。
- 服务端只比较最近 3 次相同局面的两个重复间隔；只有两个间隔相同，才视为同一重复循环。
- 重复循环：指同一局面且轮到同一方走再次出现，并且最近两段重复间隔相同。

当前服务端按以下顺序裁定：

- 先检查长将。
- 若不属于长将，再检查长捉。
- 若不满足单边长将或单边长捉，则该重复局面判和。

长将的当前实现口径：

- 长将：在同一重复循环中，如果某一方自己的每一步都对对方将/帅形成将军，则该方判负。
- 若该循环中红方自己的每一步都形成将军，而黑方不满足同样条件，则判“红方长将负，黑方胜”。
- 若该循环中黑方自己的每一步都形成将军，而红方不满足同样条件，则判“黑方长将负，红方胜”。

长捉的当前实现口径：

- 长捉：在同一重复循环中，如果某一方自己的每一步都未将军，但都持续威胁吃同一个对方非将/帅棋子，则该方判负。
- 将/帅不作为长捉目标参与长捉判定；针对将/帅的持续攻击按长将处理。

普通重复的当前实现口径：

- 普通重复：若重复循环不满足单边长将或单边长捉，则判和。

当前实现边界说明：

- 当前长捉判定只基于“持续威胁吃同一个非将/帅棋子”，不区分更细的专业规则类别。
- 如果双方在重复循环中都不构成单边违规，当前实现按和棋处理。
- 当前长将 / 长捉口径为服务端现实现定义，client 不应自行按其他象棋规则书重新裁判。

## 8. 最小示例

### 8.1 发送请求示例

红方兵 `id=29` 从初始位置 `(6,4)` 向前移动一步到 `(5,4)`：

```text
客户端 -> 服务端: [29][5][4]
```

### 8.2 成功后的返回与同步示例

```text
服务端 -> 发送方: [1]
服务端 -> 对手方: [29][5][4]
```

含义：

- 发送方收到 `1`，说明这一步成功。
- 对手方收到 `[29][5][4]`，说明需要在本地同步这一步。

### 8.3 终局通知示例

例如黑方累计 3 次违规后，服务端会继续向双方发送：

```text
服务端 -> 红方: [0xFD][1]
服务端 -> 黑方: [0xFD][1]
```

含义：

- `1` 表示红方胜。
- 收到终局通知后，client 应停止发送普通走子请求，并以服务端结果为准结束本局。
