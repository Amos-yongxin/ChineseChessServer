# 棋局操作自动化测试用例

本文档面向开发者，用于配合 `client/auto_test_client.py` 对网络对战 Server 做棋局操作自动化测试。

## 1. 前置条件

运行脚本前，请先手动完成以下操作：

1. 启动图形程序。
2. 进入“网络对战”。
3. 选择“服务器”。
4. 选择监听 IP 和 Port。
5. 点击“监听”，确保服务端已经开始接收连接。

注意：

- 每次服务端启动后，只跑一个测试套件。
- 跑下一个测试套件前，请手动重新打开一局新的服务端监听。
- 当前服务端只接受前两个连接，因此不要在同一局里反复重跑不同套件。

## 2. 运行方式

列出全部测试套件：

```bash
python client/auto_test_client.py --suite list
```

运行单个套件：

```bash
python client/auto_test_client.py --host 127.0.0.1 --port 12345 --suite smoke
```

输出 JSON 报告：

```bash
python client/auto_test_client.py --host 127.0.0.1 --port 12345 --suite perpetual_check --json-out perpetual_check.json
```

参数说明：

- `--host`：服务端监听地址，默认 `127.0.0.1`
- `--port`：服务端监听端口，默认 `12345`
- `--suite`：要运行的测试套件
- `--timeout-ms`：socket 超时，默认 `1000`
- `--strict-exploratory`：将 exploratory 套件中的异常视为失败
- `--json-out`：将结果写入 JSON 文件

## 3. 报文与判定范围

脚本当前可验证的服务端行为包括：

- 1 字节 ACK：`1` 成功，`-1` 失败
- 3 字节对手同步：`[id][row][col]`
- 2 字节终局通知：`[0xFD][result]`

终局结果码：

| 结果码 | 含义 |
| --- | --- |
| `1` | 红方胜 |
| `2` | 黑方胜 |
| `3` | 和棋 |

补充说明：

- `horse_leg`、`elephant_eye`、`self_check`、`repetition_draw`、`perpetual_chase` 使用标准开局局面。
- `perpetual_check`、`stalemate_or_no_legal_move` 依赖服务端内置的测试预设局面。
- 这些测试预设仅供开发测试使用，不属于对外公开协议。

## 4. 重复局面相关规则说明（对应当前实现）

以下 3 个套件验证的是当前服务端实现口径，不是抽象竞赛规则：

- `repetition_draw`：验证“重复 3 次且不满足单边长将 / 长捉时判和”。
- `perpetual_check`：验证“单边每步将军的重复循环判负”。
- `perpetual_chase`：验证“单边每步持续追捉同一非将 / 帅目标的重复循环判负”。

当前实现的重复循环定义：

- 重复循环：指同一局面且轮到同一方走再次出现，并且最近两段重复间隔相同。
- 判断依据包含“轮到哪一方走”，不是只看棋子摆放。
- 终局发生在“第三次形成同一循环局面”之后。
- 若测试时改动走子顺序或循环长度，预期结果可能不再成立。

## 8. 测试脚本说明

`client/auto_test_client.py` 是一个面向当前 TCP 对战协议的自动化测试客户端，主要用途如下：

- 用两个 socket 连接分别模拟红方和黑方。
- 自动发送走子请求，并校验发送方 ACK。
- 自动接收对手同步消息，并校验同步内容是否与预期一致。
- 自动接收终局通知，并校验红胜/黑胜/和棋结果。
- 在本地维护一份棋子位置与存活状态镜像，用于校验吃子后的状态变化。
- 对确定性套件输出 `PASS` / `FAIL`，对探索型套件默认输出 `OBSERVE`。

脚本特点：

- 只依赖 Python 3 标准库，不需要安装第三方包。
- 不自动启动 GUI Server，需要先手动打开并开始监听。
- 默认按“一个服务端会话只跑一个套件”设计。
- ACK、同步包、终局包都按精确字节数处理。
- 终局判断依赖服务端发出的 `[0xFD][result]`，脚本不是自行推导胜负。
