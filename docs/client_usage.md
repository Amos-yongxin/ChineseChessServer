# 终端客户端使用说明

本文档说明 `client/client.cpp` 这个 Windows 终端客户端的编译、启动和联调用法。协议细节请同时参考 [network_server_integration.md](./network_server_integration.md)。

## 1. 编译 EXE

当前终端客户端使用原生 Winsock，不依赖 Qt。

编译命令：

```powershell
g++ client\client.cpp -o client\client.exe -lws2_32
```

如果 `client.exe` 正在运行，重新编译时可能会因为文件被占用而失败；这时请先关闭旧进程，或临时输出到其他文件名。

## 2. 启动方式

命令格式：

```powershell
client\client.exe <r|b> <server_ip> [port]
```

帮助命令：

```powershell
client\client.exe --help
```

参数说明：

- `r|b`
  - `r` 表示声明本端为 red。
  - `b` 表示声明本端为 black。
- `server_ip`
  - 服务端 IPv4 地址，例如 `127.0.0.1`。
- `port`
  - 可选 TCP 端口，默认 `12345`。

示例：

```powershell
client\client.exe r 127.0.0.1
client\client.exe b 127.0.0.1
client\client.exe r 192.168.1.10 12345
```

## 3. 角色握手规则

终端客户端不会再按“先连接的是红方，后连接的是黑方”来推断角色。

现在的行为是：

- 命令行里的 `r` / `b` 会直接作为正式角色声明发给服务端。
- 连接成功后，客户端首先发送 `[0xFB][role]`。
- 只有收到服务端返回的 `[0xFB][1]` 或 `[0xFB][2]` 后，才进入对局循环。
- 如果服务端返回 `0xFE`，表示该角色已被占用。
- 如果服务端返回 `0xFF`，表示声明值非法。
- 如果服务端返回 `0xFD`，表示同一连接重复声明角色。

联调时请提前约定：

- 一端使用 `r`。
- 另一端使用 `b`。
- 不能两个客户端都用 `r`，也不能两个客户端都用 `b`。

## 4. 本地联调步骤

### 4.1 启动服务端

先启动 Qt 服务端，确认监听 IP 和端口正确，例如 `127.0.0.1:12345`。

### 4.2 启动两个终端客户端

本机联调示例：

```powershell
client\client.exe r 127.0.0.1
client\client.exe b 127.0.0.1
```

局域网联调示例：

```powershell
client\client.exe r 192.168.1.10 12345
client\client.exe b 192.168.1.10 12345
```

如果服务端已经有 red，再启动一个 `r`，第二个客户端会在握手阶段被拒绝并退出。

## 5. 终端交互方式

输入提示如下：

```text
input move (id row col, q quit):
```

规则：

- 走子格式固定为 `id row col`。
- 输入 `q` 或 `quit` 可主动退出客户端。
- 空输入、非数字、字段数量不对、范围不合法时，客户端会直接提示并要求重输，不会发送垃圾包。

示例：红方兵 `id=29` 从 `(6,4)` 走到 `(5,4)`：

```text
29 5 4
```

## 6. 终端输出说明

客户端会输出这些关键日志：

- 连接目标与本端角色。
- 角色声明是否发送成功。
- 角色握手成功或失败原因。
- 当前是否轮到自己。
- 发送请求内容，例如 `send: 29 5 4`。
- ACK 结果，例如 `ack: 1` 或 `ack: -1`。
- 违规累计，例如 `violations: 1/3`。
- 对手同步步，例如 `sync: 13 4 4`。
- 终局结果，例如 `game over: red_win`。

## 7. ACK 与终局行为

客户端的状态机规则如下：

- 收到 `ack: 1`
  - 本步成功。
  - 切换到等待对手同步。
- 收到 `ack: -1`
  - 本步失败。
  - 本地违规数加 1。
  - 仍然保持当前是自己回合，直接重新提示输入。
- 若本步直接导致终局
  - 当前发送方会在 ACK 之后继续收到 `[0xFD][result]`。
  - 被动等待的一方也可能不先收到 `sync`，而是直接收到 `[0xFD][result]`。
- 若在 ACK 位置读到不是 `1 / -1` 的值
  - 客户端会判定为“协议状态错位/非法 ACK”并退出。
  - `ack=25` 这类情况通常说明把同步包的首字节误读成了 ACK。

## 8. 常见错误与排查

### 8.1 `connect failed`

常见原因：

- 服务端没有启动监听。
- `server_ip` 填错。
- 端口不一致。
- 本机联调时没有使用 `127.0.0.1`。

### 8.2 `role handshake failed: requested role is already occupied`

说明当前角色冲突。

排查方式：

- 一端改为 `r`。
- 另一端改为 `b`。
- 确认本局里 red / black 只有一个。

### 8.3 `unexpected ack value: 25`

这不是合法 ACK。

含义通常是：

- 客户端状态机与服务端协议阶段错位。
- 本该读取 ACK 时，实际读到了同步步的首字节 `id=25`。

当前版本已经在握手阶段显式声明角色，并在 ACK 位置加入合法值校验；如果再次出现，应优先检查是否还有旧版服务端或旧版客户端混用。

### 8.4 收到 `ack: -1` 后卡住

当前版本已经修复这个问题：

- 自己收到 `ack: -1` 后不会切换到等待对手，而是继续保留自己回合。
- 如果这是第三次违规，对局会继续收到终局包并立即退出。
- 对手方在第三次违规终局时，也能直接识别 `[0xFD][result]`，不会再傻等 3 字节同步包。

## 9. 协议参考

当前终端客户端遵循的协议为：

- 角色声明：`[0xFB][1|2]`
- 角色握手应答：`[0xFB][status]`
- 走子请求：`[id][row][col]`
- ACK：`1 / -1`
- 同步步：`[id][row][col]`
- 终局通知：`[0xFD][result]`
